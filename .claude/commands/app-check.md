APP 的使用者是視障者。每次新增或修改任何功能，必須以「閉上眼睛能不能用」為標準審查。

## 修改前：先問這三個問題

1. **這個功能，使用者聽得到嗎？**
   - 操作結果必須有語音或震動回饋，不能只靠畫面變化
   - 例：按下導航按鈕 → 必須播報「開始導航」，不能只改按鈕顏色

2. **這個 UI 元素，視障者需要嗎？**
   - 純裝飾性的圖示、動畫、分隔線 → 不要加
   - 每個新增的 Widget 都要問：「拿掉它，使用者會少得到什麼資訊？」

3. **TalkBack 能讀到嗎？**
   - 所有可互動元素（按鈕、輸入框、開關）必須有 `Semantics` label
   - 圖示按鈕不能只有 icon，必須有文字描述

## 修改後：必做檢查

**語音回饋確認**
```bash
# 搜尋有沒有新增互動元素但缺少語音回饋
grep -rn "onPressed\|onTap\|GestureDetector" Android/lib/ | grep -v "//.*onPressed"
# 對照確認每個互動都有對應的 audio/tts 呼叫
```

**Semantics 標籤確認**
```bash
# 找出 IconButton 有無 tooltip 或 Semantics 包裹
grep -rn "IconButton\|Icon(" Android/lib/ --include="*.dart" | grep -v "semanticLabel\|Semantics\|tooltip"
```

**不必要內容確認**
- 新增的 Widget 是否純視覺裝飾？若是，刪掉
- 是否有只用顏色區分狀態的地方？（色盲/視障都無法識別）

## 視障者 UI 設計規則

| 規則 | 說明 |
|------|------|
| 觸控目標 ≥ 48×48 dp | 手指定位困難，按鈕要夠大 |
| 所有狀態必須播報 | 連線中、導航開始、錯誤都要有聲音 |
| 減少層級 | 畫面不能超過 2 層點擊才到目標功能 |
| 錯誤訊息必須朗讀 | 不能只顯示紅字 |
| 設定選項越少越好 | 選項太多視障者難以瀏覽 |

## 每次新增功能的自問清單

- [ ] 使用者閉眼能知道這個功能在做什麼嗎？
- [ ] 操作成功/失敗都有音效或語音嗎？
- [ ] 這個 Widget 如果拿掉，視障者會失去什麼資訊？
- [ ] TalkBack 掃描到這個元素時，說的話有意義嗎？
